## 技術メモ（ハマりどころ）
- **HEIC**: 拡張子か MIME type で判定、heic2any で変換してから IndexedDB へ
- **EXIF orientation**: サムネ生成時に補正、これを怠ると横倒し表示
- **サムネ運用**: 表示は常にサムネ。フル画像は拡大時のみ読む
- **性能**: 100〜300枚程度なら全部サムネ並べても問題ない想定
- **「近傍」のチューニング**: 最初の定義は鉛筆書き。触ってみて違和感あれば調整
- **Web制約**: 選んだ写真への永続参照は持てない → blob を IndexedDB にコピーする方式（理解の上で採用）
- **iOS Safari**: ESM 動的 import や `display:none` した file input は不安定。UMD + 可視 input が安全（v2 教訓）

## スタック
exifr / heic2any / canvas（サムネ＋orientation補正） / IndexedDB / プレーンHTML+CSS+JS
**Leaflet・地理院タイルは不要（地図は今回作らない）。バックエンドなし。**

## 完成の定義（この spike のゴール）
開く → 3枚出てくる → 1枚タップ → 時空の近くの写真が並ぶ → そのうちの1枚をタップ → さらに連想で広がる → 別の1枚でもまた歩ける → 閉じて開き直しても保持されている。
**この reminiscence walk が手で動かせる = ゴール。**

## 検証したい問い（作った後に自問する）
- 引いた3枚を見て、**気持ちいいか**（reminiscence の仮説）
- 連想で歩いていると、**やめにくいか**（associative walk の手触り）
- 「次どこ行く？」のヒントに**実際になるか**（旅の計画の軸）
- 悪い写真がランダムに出てきて気分が落ちたか／何回起きたか（curation 必要量の signal）

## Phase 1.5 — 色彩パレットによるトーン展開（AI なし・超軽量）

**目的:** 「夕焼け→別の夕焼け」「緑（苔・森）→別の緑」のような **画面のトーンによる連想ウォーク** を最速で検証する。

### 取り込み時の処理拡張
- [ ] サムネ生成時 (canvas) に簡易的な色彩特徴量を計算 (4×4 グリッドの平均 RGB = 48 次元 Float32Array)
- [ ] IndexedDB スキーマに `color` フィールドを追加して保存
- [ ] 既存写真は起動時に **非同期バックフィル** (1枚ごとに低優先度で実行、メイン操作は止めない)

### 連想展開のアップデート
- [ ] 時空近傍 (±3〜7日、5km以内) で上位 30 枚まで広めに候補抽出
- [ ] **「カラー優先」モード時**は、その 30 枚を中心写真との色距離 (ユークリッド) でソート → 上位 6 枚
- [ ] explore 画面に **「時空 / カラー」トグル** を置き、即切り替え

### 検証したい問い
- 「色の近傍」と「時空の近傍」、どちらが reminiscence をより刺激するか？
- 「夕焼け→夕焼け」「緑→緑」のような色ジャンプが気持ちいいか？

## Phase 1.8 — プログレッシブ・インデキシング（バックグラウンド非同期パイプライン）

**目的:** 大量 (数百〜数千) 投入時に「待ち時間ゼロ」でメイン体験に入る。ユーザーが遊んでる裏でじわじわインデックス化。**Phase 2 (CLIP) の前提条件**。

### アーキテクチャ
```
投入 1000枚
  ├─ フェーズA (最速): 先頭 5〜10 枚を並行処理 → 完了即メイン画面遷移
  └─ フェーズB (バックグラウンド): 残り 990 枚を1枚ずつ低優先度で処理
        ※ 1枚処理ごとに setTimeout(300ms) か requestIdleCallback で隙間
        ※ 1枚完了で IndexedDB.put、連想プールがリアルタイム拡大
```

### TODO
- [ ] インポート・キューと進捗ステート
- [ ] フェーズA: `FAST_TRACK_COUNT` 枚 (5〜10) を最速並行処理 → 完了で random 画面へ強制遷移
- [ ] フェーズB: 残りを低優先度ループ、`setTimeout(300ms)` か `requestIdleCallback`
- [ ] 控えめな進捗インジケータ (`🌀 12 / 1000` レベル、完了で消える)

### ハマりどころ
- IDB 読み書きはトランザクション安全に (バックグラウンド中もユーザー操作と競合させない)
- `URL.createObjectURL` は1枚処理ごとに必ず `revokeObjectURL` (1000枚処理後のメモリ不足クラッシュ防止)

## Phase 2 — ローカル AI (Transformers.js / CLIP) による意味展開

**目的:** 外部サーバなし、端末 WebGPU/WASM だけで「同じもの (猫、山、食べ物)」を認識して **意味空間でワープ**。

### スタック
- `@huggingface/transformers` (Transformers.js)
- モデル: `Xenova/clip-vit-base-patch32` (もしくはより軽量な MobileNet、〜100MB)
- 初回起動時に Cache Storage (or OPFS) に永続キャッシュ
- 起動時インジケータ「AIモデル準備中 (初回のみ)…」

### 取り込みパイプライン (フェーズB 内)
- サムネ → `pipeline('image-feature-extraction')` で 512 次元 Float32Array を抽出
- IndexedDB スキーマに `embedding` を追加

### セマンティック近傍ウォーク
- カードタップ時、中心 embedding と全件 embedding のコサイン類似度を一括計算
- 上位 6 枚を「意味の近傍」グリッド展開
- explore のトグルに「意味」モードを追加 (時空 / カラー / 意味)

### ハマりどころ
- **直列実行禁止** — 100枚以上のときフリーズ。`Promise.all` で 5〜10 枚チャンク
- **WebGPU フォールバック** — 使えない環境は WASM (CPU) に自動切替を transformers.js オプションで明記
- **メモリ管理** — 上同様

### 検証したい問い
- 「色」vs「意味」、どちらが脳の報酬系に効くか？
- AI の意味ワープが完璧すぎると「標準フォトアプリの AI アルバム」と既視感が出ないか？
  → あえて時空や色のノイズを混ぜたほうが連想がドライブするか？
