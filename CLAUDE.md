## 🧭 セッション開始時にまず読む

実装や方針の作業に入る前に **[DOCMAP.md](DOCMAP.md)** (資料マップ) を読み、資料構成と役割を把握する。要点だけ:
- **プロダクト化の方針 = Notion** (WHY/WHAT と HOW の 2 ページ)。**spike の検証アーカイブ = この git** (TODO / CHANGELOG / README / index.html)。
- **プロダクトのコア = 「偶然よみがえった久しぶりの記憶」** (3 要素: 偶然 / 久しぶり / よみがえる)。機能・広告・AI・名前で迷ったら、この 3 要素を強めるか / 壊すかで判断。名前は 🇬🇧 Madeleine / 🇯🇵 写真思い出。
- 「どこに何があるか / 迷ったらどこを見るか / 更新ルール」の詳細は DOCMAP.md。

## このプロジェクトの性質

- **元 spike → 製品本体（2026-06-17 統合）**。reminiscence の手触りは v75 まで検証済み。これからは Capacitor で **native 化 → App Store 申請**。「捨ててよい」段階は脱したが、**体験ロジックは資産**でそのまま乗る。整理（web ハック撤去・CDN vendoring・SQLite 化）は native の文脈で段階的に（一気に綺麗にしない）。
- ローカル完結 (写真は端末から出さない、IndexedDB→将来 SQLite)、バックエンドなし。
- スコープ境界は README.md と TODO.md の冒頭を参照。

## 🔁 セッション終了時の必須ルール

**実装作業をした session を閉じる前に、必ず TODO.md と CHANGELOG.md を更新する。**

これは次セッション (別の Claude / 別の日の自分) が状況復旧するための引き継ぎコスト最小化。spike だからこそ口頭の文脈が失われやすく、文書が唯一の memory になる。

### TODO.md の更新内容

- **「現在地」セクションを書き換える** — 何が ✅ クローズで、次の ▶ 着手前は何か、を一目で示す
- 完了したチェックボックスに `[x]` を入れる
- 触っているうちに**新しく見えてきた道**は TODO の末尾 or 「Phase X」セクションに追記
- ユーザーが「これはやらない」と言ったものは「今は作らない」リストに移す

### CHANGELOG.md の更新内容

新しいバージョン (`vN`) を**先頭に追加** (最新が上)、以下の構造で記述:

```markdown
## vN — 一行サマリ (YYYY-MM-DD)

**背景**
- なぜこの作業に着手したか、何を確かめたかったか

**設計判断**
- 採用した方針と、その理由
- 採用しなかった代替案 (あれば)

**ハマったところ** (あれば)
- 症状 → 原因 → 対処の順で具体的に
- 次の Claude が同じ罠を踏まないように

**結果 / 観察**
- 何が動いたか、ユーザーのフィードバック (生の言葉が強い signal)
- 触ってみて分かった signal

**教訓**
- このプロジェクト固有の学び (一般論ではなく)

**残課題 / 次の方向**
- 持ち越しタスク、次セッションで考えること
```

### 何を書かないか

- diff や行番号レベルの細かい変更内容 (git log で見られる)
- ありきたりな一般論 (spike 固有の学びだけ)

## 🏷 BUILD バージョン文字列の運用

`index.html` 上部に `const BUILD = 'phaseN.M ...';` がある (キャッシュ問題・ビルド切り分け用)。

- **commit ごとに必ず上げる** (例: `phase1.10` → `phase1.11`)
- **最新5版だけを保つ (v214 で導入)**: BUILD 文字列は ℹ️ 情報モーダルに全文表示するため、`(vN：一行要約｜vN-1：一行要約｜…) · 日付` の形で**最新5版・各1行**に保つ。commit ごとに先頭へ新版を1行足し、**6版目以降は削る**（過去の詳細は CHANGELOG にある）。以前は全版の詳細を連ねて 21KB 超に肥大し、情報モーダルが読めずファイルも重かった。
- iOS Safari / WKWebView の HTML キャッシュは強烈。版が変わったか確認できると「新しい版が来てない」とすぐ気付ける
- **表示場所 (v108 で変更)**: 製品化に伴い**ヘッダ表示はやめ、ℹ️ 情報モーダルの末尾＋起動時 `console.log('BUILD:', BUILD)` に移動**した (スクショ・本番のヘッダをすっきりさせるため)。ビルド確認は ℹ️ を開くかコンソールで。**ヘッダに版文字列を再表示しないこと**。
- スクショ用: web を `?shot` 付きで開くと `SHOT_MODE` が立ち、位置 UI (🛰️ ボタン・地図の軌跡トグル) を隠して native v1 (位置オフ) 見えにできる (App Store スクショを提出物と一致させる用)。

## 🚀 commit → push まで基本やる (デフォルト)

実装が一段落したら、**毎回明示依頼を待たずに commit → `git push` まで通す**。

理由: spike を GitHub Pages (`https://yutsutke.github.io/photo-memory-spike/`) で実機 (iPhone Safari) 確認する運用。push されないとスマホで触れない = push がワークフローの一部。

順序:
1. TODO.md / CHANGELOG.md 更新 (上述の必須ルール)
2. BUILD バージョン文字列を上げる (上述)
3. `git add` (個別ファイル指定、`-A` は避ける) → `git commit` → `git push` を一気通貫

コミットメッセージは日本語、既存スタイル (`feat(scope): ...` / `fix(...)` / `docs(vN): ...` / `change(...)`) に合わせる。

**例外 (明示確認が必要)**: force push、`reset --hard`、ブランチ削除、`--no-verify` などの破壊的・規約外操作。通常の main への push は確認不要。

## 🚨 iOS Safari のハマりどころ (過去にハマった)

- **HTML キャッシュ強烈**: HTML に no-cache メタタグ必須。URL `?v=N` で強制リフレッシュも併用
- **`createImageBitmap` が TypeError を投げるバージョンがある** — 古典 `<img>` + `<canvas>` + `URL.createObjectURL` の方が堅い
- **同じ名前の関数を二つ書くと事故** (`makeThumb` 衝突) — 別責務には別名
- **canvas/Blob 系のメモリ管理が緩い** — 大量の `getImageData` で表示中の Blob URL が無効化される。縮小描画でメモリ最小化、完了時に URL キャッシュ revoke + 再描画
- **フォトピッカーは HEIC を裏で JPEG 変換**。HEIC のまま渡したいなら「ファイル」アプリ経由
- **`e.target.value = ''` を await の前に呼ばない** — File 参照が無効化される疑い、`finally` で

詳細は CHANGELOG v3-v5 を参照。
